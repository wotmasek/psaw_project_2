# AI_RULES

## Jak ma pracować agent
- Najpierw czytaj dokumenty, potem koduj.
- Nie zgaduj, jeśli informacja jest już w repo.
- Nie dodawaj funkcji spoza MVP.
- Nie rozbudowuj rzeczy bez potrzeby.
- Zmieniaj najmniejszy możliwy zakres.
- Trzymaj spójność z istniejącą strukturą.

## Źródło priorytetów
Jedna obowiązująca hierarchia dokumentów jest utrzymywana w:
- `docs/README.md`

Nie powielaj tu własnej listy priorytetów.

## Zasady działania
- pracuj według zakresu MVP,
- trzymaj się ustalonej architektury,
- zachowuj prosty flow,
- nie wprowadzaj zbędnych abstrakcji,
- nie twórz dodatkowych warstw, jeśli nie są potrzebne,
- pilnuj czytelności kodu i dokumentów.

## Gdy tworzysz UI
- generuj wszystkie wymagane ekrany,
- nie pomijaj widoków,
- trzymaj nazwy zgodne z dokumentacją,
- nie mieszaj stylu z funkcją,
- nie robisz „ładnie na skróty”.

## Gdy tworzysz backend
- dbaj o spójność z modelem danych,
- pilnuj autoryzacji i ról,
- nie omijaj walidacji,
- nie upraszczaj bezpieczeństwa ponad rozsądek.

## Gdy pracujesz ze szkieletem repo
- odróżniaj plan techniczny od stanu implementacji,
- sprawdzaj `docs/tech/IMPLEMENTATION_STATUS.md` przed uznaniem czegoś za gotowe,
- jeśli kod zawiera placeholder lub TODO, traktuj to jako lukę implementacyjną, nie jako domknięty moduł.

## Gdy masz wątpliwości
- wróć do dokumentów,
- wybierz prostsze rozwiązanie,
- nie dokładaj nowych koncepcji bez potrzeby.

## Reguła końcowa
Jeśli coś nie wynika z dokumentów albo nie pomaga w MVP, nie buduj tego.
